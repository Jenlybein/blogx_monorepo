前端交互上怎么做才顺手
选中文本弹出的菜单要非常轻，最多三四个动作，不要做成二级菜单迷宫。点击后最好先弹一个小 loading，再展示“替换内容 / 追加建议 / 保留原文”三种操作。对诊断类结果，不建议自动替换原文，而是旁边列出问题卡片，点击卡片时高亮原句。整篇评分则更像右侧面板，展示“总分 + 维度雷达图/条形图 + 主要问题 + 一键跳转定位到段落”。如果能定位到段落，会比只有文字报告强很多。


[P1] 统一文章读取已经完全押注 ES，但写路径还没做到“任何会影响展示的变更都同步 ES”，这会直接造成读写不一致。文章更新只有“标签变更”或“正文变更”才触发 ES 刷新，article_update.go (line 39)；但实际会改动的字段远不止这些，比如标题、摘要、分类、封面、评论开关、状态都在更新集里，article_write_service.go (line 104)。而统一读取入口所有场景都直接查 ES，article_search.go (line 15) article_search.go (line 167)。再加上创建文章时只有“有标签”才显式刷新一次 ES 标签，article_create.go (line 40)。这意味着“改标题但不改正文”“改状态但不改标签”“无标签新建文章”都很容易出现列表页和后台页短时间甚至长时间读到旧数据。

[P1] API 进程仍然同时承担 HTTP、MySQL->ES river、图片引用 river、定时任务，部署正确性还在靠约定维持。main.go (line 13) 里还是把这些角色串在一个进程里启动，main.go (line 36)。这块你上次说先不拆，我同意可以后放，但它今天依然是整个后端最大的结构债：扩容、灰度、故障隔离、消费者幂等都会被它牵着走。


[P1] 邮件发送这一层现在有真实正确性问题：SMTP 发送失败时，函数仍然返回 nil。enter.go (line 43) 里 e.Send(...) 出错后只是 fmt.Println(e)，最后无论如何都 return nil。这会让注册、登录、重置密码这类流程把“邮件没发出去”误判成“发送成功”，这个优先级其实很高，建议尽快修。


[P2] 写路径的副作用边界还不稳定，DB、缓存、消息仍然容易分叉。评论创建是典型例子：先写评论，再改审核状态，再起 goroutine 写站内消息，再回写 Redis 计数，comment_create.go (line 72) comment_create.go (line 85)。消息服务本身又直接用全局 DB 写表，article_message.go (line 12)。文章删除虽然已经从 hook 迁出来了，但仍在事务体内直接改 Redis，delete.go (line 18) delete.go (line 84)。这类“主事务 + 最佳努力副作用”以后很适合收成显式领域服务 + post-commit/outbox。


[P2] 聊天 WebSocket 入口还是一个过大的“超级处理器”，而且同源校验是全放开的。CheckOrigin 现在直接 return true，chat_ws.go (line 32)。ChatWsView 同时负责鉴权回退、票据消费、连接生命周期、收件人校验、限流占位、消息落库、在线路由和错误回写，chat_ws.go (line 63)。这会让聊天功能一旦继续长大，测试、扩展 ack/retry、引入审核/风控都会越来越难。
