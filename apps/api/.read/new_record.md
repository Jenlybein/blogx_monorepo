## 1. 文档说明与项目定位

### 1.1 这份文档在解决什么问题

旧版开发文档很有价值，但它同时混合了：

- 开发记录
- 需求说明
- 部署步骤
- 设计设想
- 当前实现边界

这些信息放在一起对“长期维护”并不总是友好。  
本重写版希望做三件事：

1. 用“当前真实系统结构”重新组织章节，而不是按最初开发顺序记流水账。
2. 优先解释已经落地的设计与运行链路，而不是先写理想化方案。
3. 在每个模块里把“为什么这样做”写出来，而不是只列接口表。

因此，这份文档更偏向：

- 新人接手时的系统导览
- 老项目继续维护时的模块设计说明
- 代码阅读前的整体认知地图

而不只是一个“接口速查表”。

### 1.2 项目是什么

这是一个基于 Gin + GORM 的单体后端项目，但它并不是“只有文章 CRUD 的博客后端”。  
当前它已经演进成一个围绕内容平台逐步增强出来的综合型后端，覆盖：

- 用户与认证
- 文章创作与审核
- 评论互动
- 收藏夹与历史记录
- 用户关系与私信
- 站内消息与全局通知
- 搜索
- AI 创作辅助
- 后台统计
- 图片资源系统
- 日志分析

从架构形态上看，它依然是**单仓库、单进程主服务**为主，没有拆成多微服务；  
但从能力分布上看，它已经把很多基础设施问题提前解决了，例如：

- 双令牌会话体系
- 结构化日志与 ClickHouse 查询
- ES 搜索读模型
- Redis 增量计数回刷
- 七牛直传与图片引用追踪
- Binlog 监听式图片引用同步

也就是说，这个项目的特点不是“极简”，而是“单体内分层增强”。

### 1.3 当前系统的核心设计哲学

读当前代码，可以总结出几条稳定的设计倾向：

#### 1. 先把主链路跑通，再逐步增强横切能力

项目一开始显然是围绕用户、文章、评论这些主业务搭骨架，但随着功能升级，逐步把下面这些能力补进来了：

- 登录风控
- 会话失效控制
- 结构化日志
- ES 搜索
- Redis 计数
- 图片资源治理
- AI 辅助

这说明项目不是一开始就按“大而全企业架构”设计的，而是按照“先交付业务，再对关键链路持续补强”的方式长出来的。

#### 2. 更偏向务实的单体分层，而不是过度抽象

当前分层很清楚：

- `router` 负责注册路由
- `middleware` 负责横切逻辑
- `api` 负责 HTTP 层收口
- `service` 负责复杂业务
- `models` 负责数据模型
- `core` 负责运行时初始化

但没有为了追求“高级感”引入复杂 IOC 容器、领域服务注册器或模块边界 DSL。  
这种取舍对当前体量来说是合理的：  
它牺牲了一部分理论上的优雅，但换来了开发速度和可理解性。

#### 3. 高并发写热点倾向于“Redis 增量 + 定时刷库”

文章点赞、收藏、浏览、评论数、标签文章数、评论回复数等，都没有每次操作直接回写数据库总量。  
当前普遍采用的模式是：

1. 业务主关系写数据库
2. 聚合计数写 Redis 增量
3. 查询时 MySQL 基础值 + Redis 增量叠加
4. 定时任务再统一刷回 MySQL

这是一种非常典型、非常务实的内容平台写扩散方案。

#### 4. 搜索、日志、AI 都被视为“增强层”，不篡夺业务真相

- 搜索靠 ES 找结果，但 MySQL 仍是业务真相
- 日志靠 ClickHouse 查分析，但应用侧仍先写结构化文件
- AI 可以给建议，但不会直接替用户越权创建分类或标签

这说明系统始终把“真实业务数据”和“增强能力”分开治理，而不是让增强能力倒过来主导主业务。

### 1.4 技术栈

| 层次 | 当前技术 |
| ---- | -------- |
| Web 框架 | Gin |
| ORM / 数据访问 | GORM |
| 主数据库 | MySQL |
| 缓存 / 会话 / 限流 / 计数 | Redis |
| 搜索 | Elasticsearch |
| 日志分析 | ClickHouse + Fluent Bit |
| 对象存储 | 七牛云 |
| 实时通信 | WebSocket |
| AI 能力 | 外部 Chat Completion 风格接口 |
| 部署 | Docker Compose |
| 配置 | YAML + 运行时内存镜像 |

### 1.5 目录结构理解

当前仓库按典型单体项目分层，而不是按微服务拆目录。主要目录作用如下：

| 目录 | 作用 |
| ---- | ---- |
| `api/` | HTTP Handler 层，负责参数绑定、权限校验后的业务收口 |
| `service/` | 业务实现层，承担复杂读写链路、外部系统接入、同步逻辑 |
| `models/` | GORM 模型、枚举、关系表、响应拼装基础结构 |
| `router/` | 所有路由注册入口 |
| `middleware/` | 鉴权、日志、验证码、审计采集、参数绑定等横切能力 |
| `core/` | 配置读取、数据库初始化、日志器初始化、运行时启动逻辑 |
| `flags/` | 命令行模式入口 |
| `global/` | 全局运行时对象容器 |
| `conf/` | 配置结构体定义 |
| `utils/` | JWT、Markdown、请求元信息、日志序列化、安全处理等工具 |
| `init/` | 部署资源、SQL、Compose、Fluent Bit、IP 库等 |
| `.read/` | 文档、阅读记录、重写文档 |

这套结构的核心优点是：

- 找路由入口很快
- 找 HTTP 层和 service 层边界很快
- 不需要先理解复杂框架才能读业务代码

缺点是：

- 包级全局依赖较多
- 部分模块之间靠共享全局资源耦合
- 随着功能越来越多，`service/` 和 `api/` 会变厚

当前项目接受了这些代价，因为它仍处在“单体增强”的合理区间。

## 2. 开发约定与通用规范

### 2.1 Git 提交约定

虽然这份文档不再照抄旧版规范表，但当前项目的提交实践依然建议遵循明确的意图分类：

- `feat`：新功能
- `fix`：修复
- `docs`：文档
- `refactor`：重构
- `perf`：性能优化
- `test`：测试
- `build` / `ci` / `chore`：构建、配置、辅助维护

真正重要的不是“有没有严格照某个表执行”，而是：

1. 一个提交尽量表达一个完整意图
2. 不要把无关修改混在一起
3. 提交信息能让后来人看懂“为什么改”

### 2.2 信息脱敏与安全意识

当前系统里已经能看到多处对脱敏的重视：

- 站点敏感配置读取时做占位符脱敏
- 审计日志对密码、令牌、Cookie、验证码统一脱敏
- 文档里也在逐步减少真实密钥和生产地址

这说明项目已经从“自己用”向“可以被长期维护”过渡。  
后续在文档、日志、截图、示例配置里，都应继续坚持这一点。

### 2.3 配置设计原则

当前配置体系采用的是：

- YAML 文件作为持久配置
- `global.Config` 作为运行时配置镜像
- 后台配置修改时，同步改内存和文件

这个设计不是配置中心，也不是纯静态配置。  
它是一种适合单体项目的中间态方案：

- 比纯写死常量灵活
- 比数据库配置中心简单
- 比“只改文件不改内存”更适合后台动态更新

### 2.4 接口响应与错误处理原则

项目当前通过 `common/res` 统一响应输出。  
核心价值不在“长什么样”，而在两点：

1. Handler 层不用重复写 JSON 样板代码
2. 成功、失败、分页、SSE 都能保持统一风格

错误处理上，当前整体倾向于：

- 参数错误尽量翻译成可读中文
- 业务错误返回业务层可理解信息
- 内部错误通过日志进一步补上下文

这种设计适合后台管理系统和内容平台：  
用户需要读得懂的错误，开发也要追得动链路。

## 3. 运行架构与启动流程

### 3.1 启动主链路

`main.go` 当前的启动顺序可以概括为：

1. 解析命令行参数
2. 记录启动配置文件路径到 `global.Flags`
3. 读取 YAML 配置到 `global.Config`
4. 初始化雪花 ID
5. 初始化日志器
6. 初始化 Redis
7. 初始化 MySQL
8. 初始化 ClickHouse
9. 初始化 Elasticsearch
10. 根据命令行进入子模式
11. 初始化站点运行时资源
12. 启动 MySQL -> ES 同步
13. 启动图片引用监听
14. 启动定时任务
15. 启动 HTTP 服务

这个顺序很值得记录，因为它解释了不少代码现象：

- 为什么很多 service 默认直接用 `global.DB` / `global.Redis`
- 为什么命令行能力也能复用同一套初始化逻辑
- 为什么某些中间件和路由代码里几乎不传入依赖对象

如果再从维护视角拆开，这条启动链路本质上由三段组成：

#### 运行时骨架准备

- 读取配置
- 解析命令行
- 初始化日志器
- 初始化 Redis / MySQL / ClickHouse / ES

这一段解决的是“进程能不能稳定活起来”。  
只要这里失败，后面所有模块都没有意义。

#### 增强依赖接线

- 初始化站点运行时资源
- 启动 MySQL -> ES 同步
- 启动图片引用监听

这一段不只是“可选增强”，而是很多业务模块的真实依赖。例如：

- 搜索与 AI 搜索依赖 ES
- 图片治理依赖 `image_ref_river`
- 审核、发文、登录等策略依赖运行时站点配置

#### 对外服务激活

- 启动定时任务
- 启动 HTTP 服务

这一段才是系统真正开始对外提供能力。  
把这三段区分开以后，排障时会更容易判断当前故障属于：

- 进程级初始化失败
- 增强能力未就绪
- 业务入口本身异常

### 3.2 命令行与 Web 的分流

当前主程序不是“只有启动 Web 服务”一种模式，而是支持多种一次性任务：

| 命令 | 作用 |
| ---- | ---- |
| `go run . -db -f settings.yaml` | 数据库迁移 |
| `go run . -es -s init -f settings.yaml` | 初始化 ES 索引 |
| `go run . -es -s article-sync -f settings.yaml` | 全量同步文章到 ES |
| `go run . -t user -s create -f settings.yaml` | 交互式创建用户 |

设计思路很明确：  
把“运维初始化能力”放在命令行，而不是放到 Web 后台里做超级管理员按钮。  
这样安全边界更清楚，也更适合部署脚本和离线运维。

这里还有一个值得记录的小设计：  
当前命令行不是做成复杂 CLI 框架，而是用最直接的“顶层 flag + 子命令”组合：

- `-db` 负责数据库迁移
- `-es` 进入 ES 运维域，再通过 `-s` 区分 `init` / `article-sync`
- `-t user -s create` 进入用户初始化域

这种方式虽然简单，但已经足够把“Web 主服务”和“离线运维动作”彻底分开。

### 3.3 路由注册结构

所有业务接口统一挂在 `/api` 下，当前总路由入口注册了：

- `site`
- `logs`
- `images`
- `banners`
- `imagecaptcha`
- `users`
- `articles`
- `comments`
- `chat`
- `sitemsg`
- `global_notif`
- `follow`
- `fans`
- `search`
- `ai`
- `data`

总路由组还统一挂了 `LogMiddleware`。  
这说明运行日志的请求访问事件，是全站 API 默认能力，而不是某些接口才手工写。

### 3.4 全局运行时对象的设计取舍

当前系统没有引入依赖注入框架，而是把核心运行时对象放在 `global/` 中：

- `Config`
- `Logger`
- `DB`
- `Redis`
- `ClickHouse`
- `ESClient`
- `ImageCaptchaStore`
- `Flags`

这样做的优点是：

- API、middleware、service 代码写起来非常直接
- 不需要层层透传依赖
- 单体项目迭代速度快

缺点也真实存在：

- 测试时需要手工准备全局状态
- 初始化顺序有隐式耦合
- 共享可变状态在理论上更不利于强隔离

当前项目接受了这笔技术债，换取了显著的开发效率。

## 4. 数据层与基础设施

### 4.1 MySQL：业务真相存储

MySQL 在当前系统里仍然是最核心的业务真相存储。  
可以按模块把主要模型概括成几组：

#### 用户与认证

- `user_models`
- `user_conf_models`
- `user_stat_models`
- `user_session_models`

#### 用户关系与主页统计

- `user_follow_models`
- `user_view_daily_models`

#### 文章主域

- `article_models`
- `category_models`
- `tag_models`
- `article_tag_models`
- `user_top_article_models`

#### 文章互动与收藏

- `article_digg_models`
- `article_view_models`
- `favorite_models`
- `user_article_favor_models`

#### 评论域

- `comment_models`
- `comment_digg_models`

#### 资源域

- `image_models`
- `image_ref_models`
- `banner_models`

#### 消息与私信

- 站内消息相关模型
- 全局通知相关模型
- 私信会话和消息相关模型

这个模型设计说明了一个事实：  
项目虽然是博客后端，但实际已经是“内容平台型”数据库，而不是两三张表就能讲清楚的系统。

如果从“为什么要这样拆表”来看，当前模型大致可以分成四个层次：

| 层次 | 代表模型 | 设计目的 |
| ---- | ---- | ---- |
| 主实体表 | `user_models`、`article_models`、`comment_models`、`image_models` | 保存业务主体 |
| 配置 / 冗余表 | `user_conf_models`、`user_stat_models` | 把低频配置和高频展示值从主表拆开 |
| 关系表 | `user_follow_models`、`article_tag_models`、`article_digg_models`、`comment_digg_models`、`user_article_favor_models` | 保存用户与对象之间的事实关系 |
| 轨迹 / 会话表 | `user_session_models`、`user_view_daily_models`、`user_article_view_history_models` | 保存带时间维度的状态变化和行为轨迹 |

以用户域为例，这种拆法尤其明显：

- `user_models` 承担身份主体、注册来源、角色、状态、登录版本
- `user_conf_models` 承担收藏公开性、关注公开性、消息开关、陌生人私信开关
- `user_stat_models` 承担主页高频展示统计
- `user_session_models` 承担设备级登录会话

这样做会增加表数量，但能显著减少“主表不断膨胀”的问题，也让后续演进更平滑。

### 4.2 Redis：短周期状态协调层

Redis 在本项目里承担的职责非常多，但总体可以归纳成 4 类：

#### 1. 认证与风控状态

- 登录失败次数
- 登录锁定状态
- 邮箱验证码发送频率
- Token 黑名单
- 会话辅助状态

#### 2. 高并发计数增量

- 文章浏览
- 文章点赞
- 文章收藏
- 文章评论数
- 评论点赞
- 评论回复数
- 标签文章数

#### 3. 上传与审核临时状态

- 图片上传任务
- 图片审核结果暂存
- 图片任务锁

#### 4. 去重与轻量统计

- 用户主页访问日去重
- 站点流量日趋势

这说明 Redis 在当前项目里不是“热点查询缓存”，而更像一层**状态协调与增量缓冲层**。

如果从 key 家族的角度理解，当前 Redis 的主要使用方式可以概括成：

| 家族 | 典型用途 | 生命周期 |
| ---- | ---- | ---- |
| 风控键 | 登录失败次数、锁定状态、验证码频率控制 | 分钟级到小时级 |
| 会话辅助键 | Access Token 黑名单、WebSocket ticket | 与 token / ticket 生命周期一致 |
| 计数增量键 | 文章点赞、评论数、标签文章数、评论回复数 | 持续存在，靠定时任务回刷 |
| 上传任务键 | 图片上传任务、上传锁、审核暂存结果 | 分钟级到小时级 |
| 去重键 | 浏览量去重、主页日访问去重 | 通常按天过期 |

这背后的设计重点不是“加缓存提速”，而是：

1. 用 Redis 抢占判重，先挡住重复行为。
2. 用 Redis 承接高频写，把聚合回写延后。
3. 用 Redis 存那些天然短生命周期、不值得长期入库的状态。

因此在这个项目里，把 Redis 理解成“短周期状态总线”会比“缓存”更准确。

### 4.3 Elasticsearch：搜索读模型

ES 当前主要服务文章搜索。  
设计上，ES 并没有被当成文章主库，而是一个专门的搜索读模型：

1. 存储适合全文检索的文章文档
2. 提供相关度、排序、高亮、分段匹配
3. 返回命中文档 ID 和搜索辅助字段

然后再由 MySQL 回表取真实业务字段，再叠加 Redis 增量计数。

这个设计非常关键，因为它解决了两个常见问题：

- 如果只靠 MySQL 做全文检索，搜索体验和扩展性都不理想
- 如果完全依赖 ES 返回业务结果，又容易让搜索索引和业务真相脱钩

当前系统选择的是“两边取长补短”的折中方案。

### 4.4 ClickHouse：日志分析库

ClickHouse 当前不存业务主数据，只存结构化日志。  
主要三张表：

- `runtime_logs`
- `login_event_logs`
- `action_audit_logs`

其定位非常明确：

- 面向后台日志查询
- 面向近实时日志分析
- 面向分页筛选和单条详情查看

当前应用并不直接同步写 ClickHouse，而是：

1. 应用先写本地 JSON 行日志文件
2. Fluent Bit 监听日志目录
3. Fluent Bit 通过 HTTP `JSONEachRow` 写入 ClickHouse

这是典型的“应用写文件，采集器异步入分析库”的链路，稳定性和可扩展性都比同步直写更好。

### 4.5 七牛云与图片资源治理

图片系统当前已经不是“multipart 上传 -> 本地保存”这种简单模式，而是围绕七牛做了完整链路：

1. 创建上传任务
2. 签发直传 token
3. 客户端直传七牛
4. 服务端再次验收对象
5. 写正式 `image_models`
6. 接七牛审核回调
7. 通过 `image_refs` 记录资源引用
8. 通过 `image_ref_river` 自动维护引用关系

这个设计很重要，因为它说明项目已经把图片从“文件上传”升级成“资源治理”。

进一步看，当前图片系统实际治理的是四种不同状态对象：

| 对象 | 存放位置 | 作用 |
| ---- | ---- | ---- |
| 上传任务 | Redis | 管理一次上传申请、完成确认、过期与回调状态 |
| 七牛对象 | 七牛云 | 真实文件内容与审核状态 |
| 正式图片记录 | `image_models` | 平台内部可被引用的正式资源 |
| 图片引用关系 | `image_ref_models` | 图片与业务对象之间的归属关系 |

只有把这四层分开，系统才能同时支持：

- 直传
- 秒传去重
- 回调验收
- 审核状态同步
- 引用治理
- 管理员删图时的安全清理

### 4.6 Binlog 监听：图片引用关系同步

`image_ref_river` 当前是一个基于 MySQL Binlog 的监听服务。  
它会监听：

- `article_models`
- `user_models`
- `banner_models`
- `favorite_models`

当这些表发生变化时，会自动重建对应图片引用关系。  
这样做的核心价值是：

- 业务模块保存文章、头像、轮播图、收藏夹时，不需要每处都手写图片引用同步逻辑
- 图片引用模型有统一事实来源
- 删除图片时可以更稳地清引用

这是一种很有工程味道的“横切能力抽离”设计。

### 4.7 核心状态对象速览

为了让后面的业务章节更好读，这里把几个贯穿全局的状态对象单独抽出来说明。

#### `UserSessionModel`

它表示“一条可续期登录会话”，而不是登录历史。  
关键字段包括：

- `refresh_token_hash`
- `ip`
- `addr`
- `ua`
- `last_seen_at`
- `expires_at`
- `revoked_at`

设计重点是把“当前会话治理”与“历史登录审计”彻底分开。

#### `UserStatModel`

它是主页展示冗余表，而不是关系事实表。  
当前保存：

- `view_count`
- `fans_count`
- `follow_count`

也就是说，关注关系和浏览去重行为仍有自己的事实来源，但主页不必每次临时聚合。

#### `UserViewDailyModel`

这张表保存“某登录用户在某一天是否访问过某主页”。  
唯一索引由：

- `user_id`
- `viewer_user_id`
- `view_date`

组成，负责数据库层面的最终兜底。

#### `ChatSessionModel`

私信会话不是双方共用一条记录，而是**双方各保留一条以自己为视角的会话记录**。  
这样才能自然表达：

- 各自未读数
- 各自删除状态
- 各自置顶 / 静音状态

#### `ChatMsgUserStateModel`

消息主表不直接承担“我把这条消息从自己列表里删掉了”这种本地态。  
因此系统单独用这张表保存用户侧消息状态，避免一方操作污染另一方视图。

#### `ImageRefModel`

这张表记录：

- 哪张图片
- 被哪类对象引用
- 属于哪个对象
- 落在哪个字段
- 在字段中的顺序位置

这正是图片治理能落地的关键模型。

### 4.8 为什么系统允许冗余字段存在

当前代码里可以明显看到很多冗余统计和缓存增量，例如：

- 用户主页浏览数
- 粉丝数 / 关注数
- 文章点赞、收藏、浏览、评论数
- 标签文章数
- 评论回复数

这不是“表设计不规范”，而是内容平台常见的务实取舍：

1. 高频读页面不能每次都做昂贵聚合。
2. 这些统计允许短时间最终一致。
3. 只要事实表还在，冗余值就是可重建的。

因此当前系统接受这种三层结构共存：

- MySQL 事实关系
- MySQL 冗余统计
- Redis 增量补偿

### 4.9 核心存储结构速查

为了让后面各业务模块更容易回忆依赖关系，这里把关键存储对象再浓缩成一张速查表：

| 类型 | 代表对象 | 主要职责 |
| ---- | ---- | ---- |
| MySQL 主实体 | 用户、文章、评论、图片 | 保存业务主对象 |
| MySQL 关系表 | 关注、点赞、收藏、标签、置顶 | 保存事实关系 |
| MySQL 冗余 / 配置表 | 用户统计、用户配置、运行时站点配置 | 承担高频展示与运行时镜像 |
| Redis 去重状态 | 主页访问、文章浏览、验证码、登录风控 | 承担短周期判重 |
| Redis 增量状态 | 点赞数、评论数、收藏数、标签文章数 | 承担热点写削峰 |
| ES 文档 | 文章搜索文档 | 承担搜索召回与排序 |
| ClickHouse 表 | 运行日志、登录事件、审计日志 | 承担后台追溯与分析 |

## 5. 通用接口、中间件与读写模式

### 5.1 响应结构

项目当前通过 `common/res` 统一输出：

- 成功响应
- 失败响应
- 列表响应
- SSE 响应

设计重点不在返回字段长什么样，而在：

1. Handler 层不手写重复 JSON
2. 业务代码只关心成功 / 失败与数据
3. SSE 与普通 JSON 也能用同一套风格表达

### 5.2 参数绑定

当前参数绑定主要通过中间件完成：

- `BindJson`
- `BindQuery`
- `BindUri`

这样做的好处是：

- handler 不用反复写 `ShouldBind`
- 参数错误在统一层翻译成中文
- handler 层逻辑更聚焦于业务本身

### 5.3 鉴权设计

鉴权当前并不是“验一下 JWT 就结束”。  
`AuthMiddleware` 的真实工作是：

1. 解析 Access Token
2. 校验签名是否合法
3. 校验是否在 Redis 黑名单
4. 校验用户是否存在且状态允许
5. 校验 `token_version`
6. 校验 `session_id` 对应会话是否有效

这个设计的价值是：

- `token_version` 负责用户级全量失效
- `session_id` 负责设备级精确失效
- Redis 黑名单负责当前 token 的立即失效

也就是说，当前认证体系已经不是简单无状态 JWT，而是“有状态会话 + 无状态令牌”的混合体系。

如果把它再拆成更清晰的四层，会更容易理解：

#### 身份声明层

Access Token 负责高频接口上的“我是谁、角色是什么、来自哪个会话”。  
这一层要求轻量、快速、适合每个请求都校验。

#### 会话续期层

Refresh Token 不直接参与业务接口鉴权，而是用来换新 Access Token。  
它背后对应数据库里的 `UserSessionModel`，所以服务端始终保留“这台设备还能不能继续续期”的控制权。

#### 全局失效层

`token_version` 解决的是“让这个用户的所有旧 token 统统失效”。  
它很适合：

- 改密码
- 邮箱找回密码
- 明确发生安全事件时的强制下线

#### 短窗口立即失效层

Redis 黑名单解决的是更短窗口的问题，例如：

- 当前设备刚刚退出
- 某个 token 刚刚被吊销

这样即使旧 token 还没自然过期，也能立即被挡下来。

### 5.4 审计采集设计

`CaptureLog` 当前支持按位采集：

- 请求体
- 响应体
- 请求头
- 响应头

并在写日志前统一做：

- 脱敏
- 截断
- JSON 安全序列化

设计重点是把“现场采集”和“业务语义描述”拆开：

- 中间件负责拿到原始现场
- 业务代码只补充动作语义、目标对象、成功失败

这比每个 handler 自己决定怎么记日志要稳定得多。

当前会显式挂审计中间件的动作，大多满足以下一种或多种特征：

- 会修改全局配置
- 会影响全站可见内容
- 会删除用户资源
- 会发布平台级通知

因此像：

- 站点配置更新
- 标签管理
- 文章审核
- 管理员删文章
- 轮播图管理
- 全局通知发布与删除

这类接口都会更倾向于接 `CaptureLog`。

### 5.5 查询模式：`ListQuery` 与 `PageIDQuery`

项目当前并没有为所有列表查询强推一套万能方案，而是按复杂度拆分。

#### `ListQuery`

适合：

- 单表列表
- 轻量筛选
- 简单预加载

常见用于：

- 图片列表
- 轮播图列表
- 标签后台列表
- 日志后台列表
- 关系列表

#### `PageIDQuery`

适合：

- 文章列表
- 收藏夹文章列表
- 这类“筛选复杂 + 排序复杂 + 还要回表预加载”的场景

核心思路是：

1. 先查本页 ID
2. 再回表查详情

这样可以避免：

- 多表 JOIN 下分页失真
- 预加载和排序互相干扰
- 大 SQL 难以维护

这套设计是当前项目非常重要的一条底层经验。

如果更直接地比较这两种模式，可以总结成：

| 模式 | 适用场景 | 优点 | 代价 |
| ---- | ---- | ---- | ---- |
| `ListQuery` | 单表或轻联表列表 | 统一、直接、开发快 | 复杂排序和回表场景会变重 |
| `PageIDQuery` | 文章流、收藏夹内容、搜索回表 | 先裁页再回表，结果更稳 | 实现复杂度更高 |

因此当前项目并没有追求“所有列表一把梭”，而是接受两套查询策略并存。

### 5.6 SSE 的使用原则

当前项目的 SSE 主要集中在 AI 模块。  
统一实现思路是：

- 设置 `Content-Type: text/event-stream`
- 设置 `Cache-Control: no-cache`
- 关闭代理缓冲
- 服务端持续向前端输出片段

SSE 目前适合的场景包括：

- 改写时边生成边显示
- AI 搜索总结时边生成边展示

项目并没有滥用 SSE，而是只在“确实有流式体验价值”的地方使用，这个取舍很健康。

### 5.7 常用中间件一览

当前系统很多共性逻辑都通过中间件沉淀，而不是散落在 handler 中。  
常用中间件可以概括为：

| 中间件 | 作用 |
| ---- | ---- |
| `LogMiddleware` | 记录请求访问日志 |
| `AuthMiddleware` | 校验登录态、会话、token 版本、黑名单 |
| `AdminMiddleware` | 校验管理员权限 |
| `BindJson` / `BindQuery` / `BindUri` | 统一参数绑定与错误翻译 |
| `CaptchaMiddleware` | 校验图片验证码 |
| `EmailVerifyMiddleware` | 校验邮箱验证码并注入邮箱上下文 |
| `CaptureLog` | 采集请求体、响应体、头部做审计 |

这样做的价值在于：

- handler 层更薄
- 鉴权和风控不容易漏接
- 参数错误、权限错误、审计采集都能保持统一风格

## 6. 业务模块：日志系统

### 6.1 模块定位

日志模块当前已经完整重构为三条并行但结构统一的日志链路：

1. 运行日志 `runtime_logs`
2. 登录事件日志 `login_event_logs`
3. 操作审计日志 `action_audit_logs`

它们的目标不同：

- 运行日志：系统发生了什么
- 登录事件日志：认证链路发生了什么
- 操作审计日志：谁对哪个对象做了什么

### 6.2 为什么要拆三类日志

如果所有日志混在一起，至少会遇到几个问题：

- 登录失败淹没在普通请求日志里
- 操作审计和调试日志混在一起难以查
- 业务安全排查和运行问题排查需要不同筛选维度

因此当前项目选择按“问题类型”拆日志：

- 运行层问题单独查
- 安全与认证单独查
- 关键操作留痕单独查

这让后台查询场景和存储结构都更清晰。

### 6.3 统一基础字段

尽管拆成了三类日志，但它们并不是各自长得完全不同。  
当前日志系统统一抽象出基础事件字段，例如：

- `event_id`
- `ts`
- `log_kind`
- `service`
- `env`
- `host`
- `instance_id`
- `level`
- `message`
- `request_id`
- `trace_id`
- `user_id`
- `ip`
- `extra_json`

这样做的意义很大：

- Fluent Bit 采集配置简单
- ClickHouse 表结构稳定
- 后续新增日志类型也能复用基础结构

### 6.4 写入链路

当前日志写入不是同步写数据库，而是：

1. 应用写 JSON 行日志文件
2. Fluent Bit 采集日志目录
3. ClickHouse 入库
4. 后台查询 ClickHouse

这是一个很成熟的工程化选择，因为日志天然是：

- 高频追加
- 读写分离
- 更适合异步汇总

### 6.5 运行日志

运行日志当前既包含：

- 服务内 `info/warn/error` 事件
- HTTP 请求访问日志
- 第三方调用、初始化过程、同步任务日志

它依赖 `logrus`，但不是纯文本输出，而是通过：

- 公共字段 Hook
- 按日期切割文件 Hook

实现了结构化输出。

请求访问日志由 `LogMiddleware` 统一记录，包含：

- 方法
- 路由
- 状态码
- 耗时
- IP
- 当前用户 ID（能解析到时）

这说明“访问日志”在当前系统中被视作一种运行日志事件，而不是单独的一套子系统。

### 6.6 登录事件日志

登录事件日志当前不只记录“登录成功”。

已覆盖事件包括：

- `login_success`
- `login_fail`
- `login_risk_control`
- `register_success`
- `register_fail`
- `token_refresh`
- `logout`
- `logout_all`

设计上，它统一写入 `login_event_logs`，然后再通过不同读视图提供：

- 管理员查看全量登录事件
- 用户查看自己的成功登录历史

这比“用户登录历史表”和“后台登录审计表”各记一份更干净。

### 6.7 操作审计日志

操作审计日志的核心目标是追溯关键业务动作。  
当前典型场景包括：

- 管理员改用户资料
- 发布 / 审核 / 删除文章
- 配置变更
- 发布 / 删除全局通知
- 图片删除
- 分类标签管理

它的设计重点不只是“记一条日志”，而是：

1. 中间件可按需采集原始 body / header
2. 进入日志前统一脱敏和截断
3. 业务层补动作名称、目标类型、目标 ID、成功失败

这种分层使得审计日志既完整，又不会在每个 handler 里复制大量样板逻辑。

### 6.8 后台查询接口

当前后台日志查询接口已经拆成三组，而不是旧版统一 `/api/logs?log_type=*`：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/logs/runtime` | GET | 运行日志列表 |
| `/api/logs/runtime/:id` | GET | 运行日志详情 |
| `/api/logs/login` | GET | 登录事件日志列表 |
| `/api/logs/login/:id` | GET | 登录事件日志详情 |
| `/api/logs/action` | GET | 审计日志列表 |
| `/api/logs/action/:id` | GET | 审计日志详情 |

用户侧还有一条特别接口：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/users/login/log` | GET | 查看成功登录历史 |

### 6.9 查询维度与默认规则

后台日志查询当前并不是只有简单分页，而是已经具备比较明确的筛选语义。

#### 运行日志

当前支持按以下维度筛选：

- 时间范围
- `service`
- `level`
- `host`
- `method`
- `path`
- `user_id`
- `key`

其中有两个实现细节很值得记录：

1. 如果没有显式传时间范围，默认只查最近 7 天。
2. `key` 会同时作用于 `message`、`path`、`request_id` 等字段，而不是只查一列。

#### 登录事件日志

当前支持：

- 时间范围
- `user_id`
- `ip`
- `username`
- `login_type`
- `event_name`
- `success`

这组筛选维度直接对应了登录排障最常见的问题：

- 某个用户最近为什么登不上
- 某个 IP 是否在批量撞库
- 某类登录方式是否异常

#### 操作审计日志

当前支持：

- 时间范围
- `user_id`
- `ip`
- `action_name`
- `target_type`
- `target_id`
- `success`

它的设计目标很明确，就是让后台能快速回答：

- 谁在什么时间对哪个对象做了什么
- 这次动作是否成功
- 失败时请求体和响应体是什么

### 6.10 写入失败与退化策略

日志系统虽然已经很完整，但当前仍然坚持一个重要原则：  
**日志链路故障不应轻易拖死主业务链路。**

这体现在几个方面：

#### 应用先写本地文件

应用不是直接同步写 ClickHouse，而是先写本地 JSON 行文件。  
这让运行时更容易承受：

- ClickHouse 短时不可用
- 网络抖动
- 分析库升级或重启

#### 采集链路异步化

Fluent Bit 负责从日志文件到 ClickHouse 的异步搬运。  
因此日志分析库出问题时，应用依然可以先把原始日志落地。

#### 查询层不回扫原始文件

后台日志查询当前明确依赖 ClickHouse，不会退回去扫应用原始日志目录。  
这是有意取舍：

- 好处是后台查询接口足够统一
- 代价是分析链路中断时，后台功能会退化

### 6.11 当前边界

- 当前查询依赖 ClickHouse，未启用时不会自动降级成扫原始日志文件
- 当前没有在线浏览原始日志目录的接口
- 当前 `trace_id` 字段已预留，但系统主链路仍主要依赖 `request_id`

## 7. 业务模块：站点配置与资源系统

### 7.1 站点配置

#### 模块定位

站点配置模块承担两类职责：

1. 对外提供公共站点配置读取
2. 提供后台敏感配置读取与更新

它管理的配置类型当前包括：

- `site`
- `email`
- `qq`
- `qiniu`
- `ai`

#### 设计思路

当前并没有为这些配置单独建一套数据库配置中心，而是继续沿用：

- YAML 作为持久配置
- `global.Config` 作为运行时镜像
- 后台更新时同步回写文件

这是一种对当前体量非常实用的方案：

- 开发和部署成本低
- 修改配置后立即影响运行时
- 不需要额外维护配置表和配置缓存一致性

#### 关键接口

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/site/qq_url` | GET | 获取 QQ 登录跳转地址 |
| `/api/site/:name` | GET | 获取公开站点配置 |
| `/api/site/admin/:name` | GET | 获取后台敏感配置（脱敏） |
| `/api/site/:name` | PUT | 更新指定配置并回写 YAML |

#### 公开配置与敏感配置的边界

当前站点配置并不是“所有配置都能直接对外读”。  
系统明确区分了两类读视图：

- 公开视图：给前台或登录流程读取必要配置
- 后台视图：给管理员读取敏感配置，但会做脱敏

例如：

- QQ 登录页只需要 `qq_url`
- SEO 页面只需要标题、图标、关键词
- 后台管理页才需要看到更完整的邮箱、七牛、AI 等配置结构

这意味着站点配置模块本身就承担了“配置最小暴露面控制”的职责。

#### 更新链路为什么要同时改内存和文件

如果后台更新配置只改文件，不改内存：

- 当前进程不会立刻生效
- 管理员会误以为更新失败

如果只改内存，不改文件：

- 进程重启后会丢失
- 多环境复制也会困难

因此当前采用“双写”：

1. 修改运行时镜像
2. 持久回写 YAML

这是一种适合单体项目的务实设计。

#### 运行时配置镜像并不只在文件里

当前项目实际上已经把“运行时站点配置”从纯文件配置进一步升级为：

- 默认值来自 `site_default_settings.yaml`
- 数据库存一份运行时配置镜像
- 内存里再缓存一份当前运行态

这意味着站点配置已经具备了“启动时灌入默认值、运行时从数据库恢复、更新后同步回内存”的完整链路。  
它仍然不是配置中心，但已经比“只读 YAML 文件”更适合后台动态管理。

#### 站点配置的实际消费方

站点配置并不是一个独立存在的“后台表单模块”，而是多个业务模块共同依赖的运行时输入：

- 注册 / 登录流程依赖邮箱、QQ、风控与站点开关
- 发文流程依赖审核相关配置
- 图片上传依赖七牛配置
- AI 模块依赖 AI 服务配置
- SEO 与公共站点信息依赖站点基础配置

### 7.2 图片资源系统

#### 模块定位

图片系统当前已经重构为资源生命周期系统，不再是简单的上传接口。  
它覆盖：

- 上传任务创建
- 七牛直传
- 服务端验收
- 内容审核
- 正式资源入库
- 图片引用追踪
- 管理员删除与清理

#### 为什么重构成任务化上传

旧式后端上传模型的问题通常是：

- 服务端吃上传流量
- 上传成功不等于资源可信
- 重复上传难治理
- 无法优雅接对象存储审核回调

当前任务化链路把上传拆成：

1. 声明
2. 传输
3. 验收

于是：

- 大文件流量直接走对象存储
- 服务端只在关键节点做校验和确权
- 重复内容可以做秒传和复用
- 审核、回调、引用关系都能纳入统一资源模型

#### 上传任务设计

当前前端上传前先调用：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/images/upload-tasks` | POST | 创建上传任务 |

上传任务只接收元信息：

- `file_name`
- `size`
- `mime_type`
- `hash`

服务端先做第一轮静态校验：

- 七牛配置是否启用
- 文件大小是否超限
- MIME 是否是图片
- 后缀是否在白名单
- 哈希是否为空

如果数据库里已有相同内容图片，则直接秒传返回已有资源信息。

#### 七牛直传与确认

上传任务创建成功后，服务端返回：

- `bucket`
- `object_key`
- `upload_token`
- 过期时间

客户端把文件直接传七牛。  
之后可以通过两种方式确认：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/images/qiniu/callback` | POST | 七牛上传回调 |
| `/api/images/upload-tasks/complete` | POST | 手动确认上传 |

确认时服务端不会信任前端，而是再次向七牛查询真实对象：

- 对象是否存在
- 哈希是否一致
- 大小是否一致
- MIME 是否仍为图片
- 格式和宽高是否合理

也就是说，真正决定“这是不是一张正式图片”的，是服务端验收，而不是上传成功事件本身。

#### 图片正式入库

正式图片会写入 `image_models`。  
其核心字段包括：

- 上传用户
- provider
- bucket / object_key
- URL
- 文件名
- MIME
- 大小
- 宽高
- hash
- status

当前 `hash` 实际保存的是七牛返回的 `etag`。  
这意味着图片去重最终以对象存储校验结果为准，而不是只信前端提交的哈希。

#### 审核状态同步

图片模块当前还接了七牛审核回调：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/images/qiniu/audit/callback` | POST | 同步审核结论 |

审核结果会映射成系统内部状态，例如：

- `pass`
- `review`
- `block`
- `deleted`
- `orphaned`

如果回调先到、图片后入库，则审核结果先暂存 Redis，等图片正式写库时再消费。  
这个小设计很重要，因为它处理了“外部回调与业务入库谁先到”的典型时序问题。

#### 图片引用关系

图片上传成功并不意味着它已经被业务使用。  
当前真正的业务引用关系由 `image_refs` 记录，核心字段包括：

- `image_id`
- `ref_type`
- `owner_id`
- `field`
- `position`

而 `image_ref_river` 会监听业务表变更，自动重建：

- 文章封面图引用
- 文章正文图片引用
- 用户头像引用
- 轮播图封面引用
- 收藏夹封面引用

这个设计把“图片上传”和“图片被哪个业务对象使用”彻底解耦了。

#### 上传任务的状态机

把图片系统理解透，关键在于理解上传任务而不是只看接口名。  
一次上传任务通常会经历这些状态：

1. 已创建，等待上传
2. 客户端直传七牛
3. 等待服务端确认
4. 服务端验收成功，转为正式图片
5. 审核回调到达，状态更新
6. 异常情况下进入失败、拦截、孤儿对象等状态

这条状态机的价值在于：

- 上传和正式入库不再是同一瞬间
- 回调早到或晚到都能被接住
- 资源治理可以围绕“正式图片”而不是“临时上传对象”开展

#### 删除链路为什么先删对象再删引用

管理员删除图片当前不是简单删数据库记录。  
更合理的理解顺序是：

1. 删除对象存储上的文件
2. 清理图片引用
3. 删除正式图片记录

这样做的原因是：

- 先删数据库记录会导致后续对象清理缺少依据
- 先删引用再删对象，若对象删除失败，可能让引用状态提前失真

这也是资源治理系统和“普通文件表”最大的差别之一。

#### `image_ref_river` 的真实价值

如果没有 `image_ref_river`，下面这些业务模块几乎都要各自维护一套图片引用同步逻辑：

- 文章内容与封面
- 用户头像
- 轮播图封面
- 收藏夹封面

那样不仅重复，而且很容易出现：

- 某一处忘记补引用
- 某一处删除时漏清理
- 不同模块解析图片 URL 的逻辑不一致

当前把它抽成基于 Binlog 的统一河流服务，等于把“引用同步”从业务模块里剥离成了横切能力。

#### 图片系统的部署前提

当前图片系统要完整工作，除了后端本身，还依赖：

- 七牛配置可用
- 上传回调地址可从外网访问
- 审核回调地址可从外网访问
- `image_ref_river.enabled` 等相关配置启用
- MySQL Binlog 权限与位点监听条件满足

这也是为什么图片模块已经不只是一个 API，而是一条跨外部服务和内部同步链路的完整系统。

#### 图片管理接口

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/images/upload-tasks` | POST | 创建上传任务 |
| `/api/images/upload-tasks/complete` | POST | 手动确认上传 |
| `/api/images/upload-tasks/:id` | GET | 查询任务状态 |
| `/api/images/qiniu/callback` | POST | 七牛上传回调 |
| `/api/images/qiniu/audit/callback` | POST | 七牛审核回调 |
| `/api/images` | GET | 管理员图片列表 |
| `/api/images` | DELETE | 管理员批量删除图片 |

#### 当前边界

- 当前只接了七牛这一种存储后端
- 普通用户侧没有“我的图片库”列表接口
- 删除接口偏管理员强制删除，不会因图片仍被引用而主动阻止

### 7.3 轮播图

轮播图模块当前相对简单，围绕三个核心字段：

- `show`
- `cover`
- `href`

设计思路是先满足首页轮播展示，不提前做复杂排序、素材审核、投放时间窗这类高级能力。  
这是一种比较克制的实现。

当前轮播图模块的接口很简单：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/banners` | GET | 轮播图列表 |
| `/api/banners` | POST | 创建轮播图 |
| `/api/banners/:id` | PUT | 更新轮播图 |
| `/api/banners` | DELETE | 批量删除轮播图 |

它虽然简单，但仍体现了两个设计点：

1. 轮播图封面同样会纳入图片引用体系，而不是独立粗暴存 URL。
2. 创建、更新、删除都接了审计日志，这说明它被视为后台运营动作，而不是随手改的小配置。

### 7.4 当前边界

- 站点配置当前仍以单 YAML 文件为主，不是多环境配置中心
- 图片系统当前只面向七牛云对象存储，没有抽象成多存储后端插件
- 轮播图还没有投放时间窗、排序权重和多端差异化投放能力

## 8. 业务模块：用户、认证与主页

### 8.1 认证体系总览

当前认证链路已经升级为一套完整体系，而不是旧版单 JWT 方案。  
大致可以拆成四层：

1. 挑战校验层
2. 风控层
3. 令牌与会话层
4. 失效控制层

#### 挑战校验层

- 图片验证码
- 邮箱验证码

#### 风控层

- 密码登录失败限流
- 邮件发送频率限制

#### 令牌与会话层

- Access Token
- Refresh Token
- UserSession

#### 失效控制层

- `token_version`
- Redis 黑名单
- 会话吊销

这套设计说明当前系统已经把认证视为一套“完整登录态治理”问题，而不是单纯发 token。

如果再从“登录一次到底发生了什么”来理解，当前统一登录结果通常包含：

1. 校验挑战条件是否通过
2. 校验用户状态是否允许登录
3. 通过风控后创建或刷新一条会话
4. 签发新的 Access Token
5. 写入新的 Refresh Token
6. 记录登录事件日志

也就是说，当前登录不是“查到用户就返回 token”，而是一次带风控、带会话、带日志的完整状态建立流程。

### 8.2 图片验证码

图片验证码当前使用 `base64Captcha`。  
它的设计思路是：

- `GET /api/imagecaptcha` 只负责生成验证码图片
- `CaptchaMiddleware` 只负责在需要的业务接口里校验验证码

这样做的好处是：

- 生成和消费解耦
- 登录、发邮箱验证码等多个高风险接口都可复用同一校验层

当前存储后端仍是进程内存，对多实例部署并不天然友好，但对当前体量足够简单直接。

从系统边界看，图片验证码当前承担的是“低成本挡住脚本化直接打接口”的作用，而不是强对抗型安全能力。  
它更适合放在：

- 密码登录
- 发送邮箱验证码

这些容易被批量尝试的入口前面。

### 8.3 邮箱验证码

邮箱验证码当前主要服务：

- 邮箱登录
- 邮箱注册
- 密码找回
- 邮箱绑定

其核心设计点是：

- Redis 存储 `email_id -> 邮箱、验证码、失败次数等状态`
- 通过 Lua 脚本做原子校验与失败计数处理
- 校验成功后删除验证码并把邮箱写入上下文

这条链路比“验证码只存在前端 session”稳得多，也更适合多次失败计数与频率控制。

当前邮箱验证码链路有几个很重要的工程特征：

- 发送和验证使用 Redis 状态而不是数据库表
- 验证成功后会消费掉验证码，避免重复使用
- 验证后的邮箱信息会进入上下文，后续接口直接复用，不必再次让前端传邮箱和验证码

这种设计让注册、登录、绑邮箱、找回密码几条流程可以共享同一套中间件能力。

### 8.4 密码 / 邮箱 / QQ 登录

当前登录入口包括：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/users/login` | POST | 密码登录 |
| `/api/users/email/login` | POST | 邮箱验证码登录 |
| `/api/users/email/register` | POST | 邮箱注册 |
| `/api/users/qq` | POST | QQ 登录 |

它们虽然入口不同，但最终都会收敛到同一类结果：

- 创建服务端会话
- 签发 Access Token
- 写 Refresh Cookie
- 记录登录事件日志

这说明当前系统已经把“登录方式差异”和“登录态建立”拆开治理。

它们之间的差异主要只发生在“身份证明阶段”：

- 密码登录依赖账号密码和图片验证码
- 邮箱登录依赖邮箱验证码中间件
- QQ 登录依赖第三方授权信息

而一旦进入“确认这个用户可以登录”的阶段，后面的会话、token、日志链路会尽量收敛成统一实现。

### 8.5 Refresh Token 与会话续期

`POST /api/users/refresh` 当前只认 Cookie 中的 `refresh_token`。  
其核心逻辑不是重复发老 token，而是：

1. 找到当前有效会话
2. 校验 `refresh_token_hash`
3. 生成新的 Access Token
4. 生成新的 Refresh Token
5. 更新会话中的 token 哈希、过期时间、最后访问时间、IP、UA、归属地

这是一种比较完整的“会话续期 + Refresh Token 轮换”设计。

这里的关键不是“又发了一次 Access Token”，而是：

- Refresh Token 每次续期都会轮换
- 会话元信息会跟着刷新
- 老的 Refresh Token 不应该长期可复用

这能显著降低 Refresh Token 泄露后的长期风险。

### 8.6 退出登录与改密码失效控制

当前支持：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/users/logout` | POST | 退出当前设备 |
| `/api/users/logout/all` | POST | 退出全部设备 |
| `/api/users/password/recovery/email` | PUT | 通过邮箱重置密码 |
| `/api/users/password/renewal/email` | PUT | 登录后改密码 |

其中失效控制的设计重点是：

- 退出当前设备：吊销当前 `session_id` + 当前 Access Token 拉黑
- 退出全部设备：吊销全部会话 + 当前 Access Token 拉黑
- 改密码 / 重置密码：事务内更新密码 + `token_version + 1` + 吊销全部会话

这个设计非常关键，因为它让“改密码后旧设备全部下线”成为可靠的系统行为，而不是靠前端配合。

从安全语义上讲，当前系统把几类动作明确区分开了：

- 退出当前设备：局部失效
- 退出全部设备：全会话失效
- 改密码 / 找回密码：全会话失效 + 全 token 版本升级

这种区分对排查用户投诉和安全事件非常重要。

### 8.7 用户资料与当前用户完整资料

当前资料模块包含：

- `GET /api/users/detail`
- `GET /api/users/base`
- `PUT /api/users/info`
- `PUT /api/users/admin/info`

`detail` 面向当前登录用户，返回：

- 用户基本资料
- 用户配置
- 是否已设置密码
- 注册来源
- 邮箱等

而 `base` 则面向用户主页首屏展示。

`detail` 和 `base` 的分工非常明确：

- `detail` 面向“我自己”，返回更完整的账号与配置视图
- `base` 面向“某个主页”，返回首屏展示所需聚合信息

这比“一个用户详情接口同时兼顾自己和别人”更利于演进。

### 8.8 用户主页基础资料与关系态

`GET /api/users/base` 当前已经不是简单“昵称 + 头像”接口，而是一条聚合型主页资料接口。  
它会一次返回：

- 用户基础资料
- 用户配置可见性
- 用户统计冗余值
- 当前访问者与主页主人的关系态

具体包括：

- `view_count`
- `fans_count`
- `follow_count`
- `favorites_visibility`
- `followers_visibility`
- `fans_visibility`
- `home_style_id`
- `relation`

这使得前端在进入用户主页时，不需要再额外发多条请求去拿统计与关系态。

除此之外，当前 `users/base` 还会返回：

- `place`

它来自用户注册或最近更新时解析出的地址信息。  
这说明主页基础资料接口已经不只是身份卡片，而是在承担“首屏资料聚合接口”的职责。

### 8.9 用户主页浏览统计设计

用户主页浏览数不是简单每次访问都 +1，而是：

- 只有登录用户访问他人主页才统计
- 访问自己主页不统计
- 游客访问不统计
- 同一访客对同一主页同一天只计 1 次

当前实现采用双层方案：

1. Redis `HSETNX` 抢占式判重
2. `UserViewDailyModel` 唯一索引兜底

并且事务内完成：

- 写日访问记录
- 更新 `UserStatModel.ViewCount`

如果事务失败，再尽力回滚 Redis 占位。  
这是一个非常典型、非常正确的“Redis 先挡板 + DB 唯一索引兜底 + 失败回滚 Redis”的设计。

### 8.10 用户统计冗余表

当前新增 `UserStatModel` 维护主页展示所需的冗余统计：

- `ViewCount`
- `FansCount`
- `FollowCount`

其中：

- `ViewCount` 由主页访问逻辑维护
- `FansCount / FollowCount` 由关注业务在事务内同步维护

这说明当前系统已经把“统计展示值”和“关系事实表”分开治理：

- 事实表负责真实关系
- 冗余统计负责高频展示

同时，`UserModel.AfterCreate` 会自动初始化：

- `UserConfModel`
- `UserStatModel`

这意味着一个用户在创建成功后，就天然具备：

- 默认隐私配置
- 默认消息配置
- 默认主页样式
- 初始统计对象

而不需要业务代码在每条注册路径里分别补建。

### 8.11 用户资料编辑

`PUT /api/users/info` 当前不是所有字段直接更新同一张表，而是会拆成：

- 用户主表字段
- 用户配置表字段

这样做的好处是：

- 主体资料与配置边界更清楚
- 新增配置项更容易扩展
- 不必在一张主表中不断堆低频配置字段

同时，用户名更新当前还有频率限制和格式校验，而不是只靠前端输入框约束。

## 9. 业务模块：文章主域

### 9.1 模块定位

文章模块是当前项目的核心业务域，覆盖内容平台最复杂的一组能力：

- 文章创建
- 文章更新
- 文章列表
- 文章详情
- 审核与删除
- 点赞
- 浏览
- 收藏夹
- 收藏关系
- 浏览历史
- 分类
- 标签
- 置顶

读这部分代码时，不能把它当普通博客文章 CRUD 看待。  
当前文章模块已经和：

- 搜索
- 消息
- Redis 增量
- 审核流
- 收藏夹
- 关系与主页展示

这些模块交织在一起。

### 9.2 文章内容存储策略

当前文章正文已经不再采用旧版：

- Markdown 原文一份
- HTML 持久化一份

的双存方案。

现在的策略是：

1. 前端提交 Markdown 内容
2. 后端通过统一安全处理逻辑做清洗
3. 把安全处理后的内容直接写入 `article_models.content`

这个设计的好处是：

- 模型更简单
- 不需要额外维护 `html_content`
- 后续展示层如果要渲染 HTML，可以基于安全 `content` 再处理

代价是：

- 如果未来确实要区分“展示态 HTML”和“编辑态原文”，需要再补专门结构

当前系统接受了这个取舍。

### 9.3 文章状态流与可见性规则

当前文章并不是只有“存在 / 不存在”两种状态，而是至少包含：

- 草稿
- 审核中
- 已发布
- 已拒绝

虽然不同接口对状态值的使用视角不同，但整体可以这样理解：

1. 作者创建时可以选择保存为草稿或提交发布
2. 如果站点配置要求审核，则提交发布后进入审核中
3. 管理员审核后变成已发布或已拒绝
4. 已发布文章才会稳定出现在公开列表、搜索和年度统计里

因此文章状态不仅影响后台展示，也直接影响：

- 前台可见性
- 搜索是否可搜
- 置顶是否能展示
- 统计是否计入

### 9.4 文章创建

`POST /api/articles` 当前负责：

- 创建文章主记录
- 处理正文与摘要
- 建立分类关系
- 建立标签关系
- 根据站点配置决定是否直接发布或进入审核

这里的设计重点是：

1. 正文与标签关系放在同一事务
2. 摘要可自动提取
3. 是否免审核由站点配置控制

这说明发文不是单纯写一条文章记录，而是内容数据的整组建模过程。

一个更完整的创建链路可以理解为：

1. 校验标题、正文、分类、标签是否合法
2. 对 Markdown 内容做安全处理
3. 生成或修正摘要
4. 根据站点规则决定文章初始状态
5. 事务内保存文章主记录与标签关系
6. 触发后续搜索同步、消息、统计等增强能力

### 9.5 文章更新

`PUT /api/articles/:id` 当前不是万能保存接口，而是：

- 只允许作者编辑自己的文章
- 重新处理正文与摘要
- 重新处理分类和标签关系
- 可能因站点审核规则而退回审核中

设计重点是：

- 更新时同样使用事务维持正文与标签关系一致
- 标签文章数增量通过 Redis 维护，不是每次直接刷数据库总量

更新和创建最大的区别不在于“是改已有记录”，而在于它还要处理：

- 原有标签关系替换
- 状态是否回退到审核中
- 原文章是否仍满足置顶展示条件

### 9.6 文章列表

文章列表是当前最复杂的查询之一。  
它同时要支持：

- 首页文章流
- 某个用户的文章列表
- 当前用户自己的文章列表
- 管理员全站文章管理列表

还要叠加：

- 分类筛选
- 标签筛选
- 状态筛选
- 标题关键字
- 多种排序
- 作者置顶
- 管理员置顶

因此当前没有直接套 `ListQuery`，而是使用 `PageIDQuery` 做两段查询：

1. 第一段只拿当前页文章 ID
2. 第二段按 ID 回表查文章详情、作者、分类、标签

这正是 `PageIDQuery` 设计存在的意义。

#### 列表的主要查询类型

当前列表接口的 `type` 至少承担了几种视角：

- 查自己文章
- 查别人的文章
- 管理员查全站文章

这意味着同一条接口既是前台内容流，又承担了作者后台和管理员后台的列表能力。  
设计上通过参数而不是拆多条路由，换来的是统一的返回结构。

#### 列表排序为什么要先查 ID

文章列表会叠加：

- 状态过滤
- 分类过滤
- 标签过滤
- 作者维度
- 置顶逻辑
- 多种排序字段

如果一开始就在完整预加载结果集上做分页，很容易：

- 页内顺序不稳定
- SQL 过重
- 预加载干扰分页结果

因此当前选择了“先取 ID，再回表”的两段查询。

### 9.7 文章详情

`GET /api/articles/:id` 当前不仅返回公共内容，还在有登录态时额外返回：

- `is_digg`
- `is_favor`

也就是说，文章详情现在已经是“公共文章内容 + 当前用户互动态”的混合读接口。

这样设计的好处是：

- 前端文章页首屏渲染时，不需要再额外查“我有没有点过赞 / 收藏过”
- 用户体验更自然

同时，文章详情不会顺手增加浏览量，浏览统计被拆到了单独接口。  
这是一个很好的职责分离：

- 详情接口优先保证稳定返回正文
- 浏览统计失败不影响正文展示

此外，文章详情当前返回的不是简化卡片，而是完整阅读视图，核心字段包括：

- 标题、摘要、正文、封面
- 浏览 / 点赞 / 评论 / 收藏数
- 评论开关
- 状态
- 标签
- 作者头像、昵称、用户名
- 分类名
- 当前用户的 `is_digg`、`is_favor`

这说明详情接口承担的是“文章阅读页首屏完整数据源”的职责。

### 9.8 文章审核与删除

审核接口由管理员使用，负责把文章从：

- 审核中 -> 已发布
- 审核中 -> 已拒绝

删除则分成：

- 用户删除自己的文章
- 管理员批量删除文章

当前删除并不只是删文章主表，而是依赖删除钩子和相关链路清理：

- 评论
- 文章点赞关系
- 收藏关系
- 置顶关系
- 浏览历史
- 标签关系

这说明文章在系统里是一个“关联中心”，删除时必须清多处状态。

审核流程当前也不是单纯改一个状态字段，而是会影响：

- 文章是否进入公开流
- 是否进入搜索索引
- 是否能被首页、分类页、统计页看见

因此审核能力实际上是文章主域和搜索 / 统计 / 展示层之间的重要闸门。

### 9.9 文章互动：点赞、浏览、收藏

文章互动能力当前采用“数据库事实关系 + Redis 计数增量”的组合设计。

#### 点赞

- 切换式接口
- 建立 / 删除点赞关系
- Redis 维护点赞增量

#### 浏览

- 单独接口统计
- 登录用户按 `user_id + article_id` 去重
- 游客按 `ip + ua` 哈希与文章 ID 去重
- 到次日 0 点过期

#### 收藏

- 不是简单 `is_favor` 布尔值
- 而是用户可把文章放进多个收藏夹

这说明当前系统对“收藏”采用的是收藏夹模型，而不是轻量单关系模型。

这里三种互动的建模方式并不相同：

- 点赞：更接近“用户与文章的布尔关系”
- 浏览：更接近“去重统计事件”
- 收藏：更接近“文章进入用户自定义容器”

这说明当前项目没有为了统一接口风格而强行把三者建成同一种模型。

### 9.10 收藏夹与文章收藏

收藏夹设计是文章模块里非常值得记录的一块，因为它体现了“先把模型设计对，再谈交互”。

当前有两张关键模型：

- `favorite_models`
- `user_article_favor_models`

这样带来的好处是：

- 一个用户可以有多个收藏夹
- 一篇文章可放入多个收藏夹
- 支持默认收藏夹
- 支持批量取消收藏
- 支持收藏夹公开性控制

当前 `GET /api/articles/favorite` 在 `type=1` 且带 `article_id` 时，还会返回 `has_article`。  
这非常适合前端弹出“收藏到哪个收藏夹”的场景，因为前端能直接知道：

- 哪些收藏夹已经有这篇文章

而不用额外逐个判断。

从返回结构看，收藏夹列表当前已经不只是返回收藏夹自身，还会补：

- 文章数量
- 在 `type=1 + article_id` 下的 `has_article`
- 某些视图中的用户昵称与头像

这说明收藏夹接口已经兼顾了：

- 用户管理自己的收藏夹
- 管理员查看收藏夹
- “把文章收藏到哪个收藏夹”这种弹窗场景

### 9.11 浏览历史

浏览历史当前也是独立能力，不是浏览量统计的副产物。  
写入时使用 `ON CONFLICT` 风格的 upsert，核心目的是：

- 幂等
- 高并发下不重复插入
- 最近看过即可更新 `updated_at`

这是非常合适的历史记录建模方式。

浏览历史和浏览数是两套完全不同的概念：

- 浏览数服务于公共热度统计
- 浏览历史服务于个人回看记录

把它们拆开之后，个人删历史并不会影响公共热度，这一点非常关键。

### 9.12 分类、标签与置顶

#### 分类

- 用户私有资源
- 支持列表、创建、更新、删除、下拉选项

#### 标签

- 全站词典
- 由管理员维护
- 发文和改文时只能引用已启用标签
- 标签文章数使用 Redis 增量 + 定时刷库

#### 置顶

项目当前支持：

- 作者置顶
- 管理员置顶

两类置顶共用同一张关系表，而不是拆成两套模型。  
这种设计很务实：

- 结构更简单
- 查询逻辑可复用
- 置顶状态变更也更容易同步到 ES 文档

再展开一点，当前置顶能力的设计重点有三层：

1. **关系表达层**  
   置顶本质上仍是“用户与文章之间的一种特殊关系”，所以适合用关系表承载。
2. **列表排序层**  
   文章列表不会只看发布时间，还会叠加作者置顶和管理员置顶。
3. **搜索一致性层**  
   置顶状态变化后会同步刷新 ES 文档，保证搜索结果与文章流排序语义尽量一致。

这说明当前系统把“置顶”当成文章展示秩序的一部分，而不只是一个简单布尔字段。

#### 分类、标签、置顶的角色差异

这三类能力虽然都与文章展示相关，但职责完全不同：

- 分类：更偏作者自己的内容组织
- 标签：更偏平台公共词典
- 置顶：更偏内容展示秩序控制

正因为角色不同，所以它们也采用了不同治理方式：

- 分类由用户私有维护
- 标签由管理员维护
- 置顶由作者和管理员共同参与，但语义不同

### 9.13 缓存同步与一致性设计

文章域是当前缓存与异步同步最重的业务域之一。  
围绕文章，系统至少会维护这些增强状态：

- 点赞数 Redis 增量
- 收藏数 Redis 增量
- 浏览数 Redis 增量
- 评论数 Redis 增量
- 标签文章数 Redis 增量
- ES 文章文档

因此文章模块的设计重点从来不只是“把文章存进 MySQL”，而是：

- MySQL 负责业务真相
- Redis 负责热点计数与短期实时性
- ES 负责搜索读模型

这也是文章域比普通 CRUD 模块明显更重的根本原因。

此外，围绕这些缓存增量，系统还会依赖定时任务或批量同步逻辑去做“最终回刷”。  
也就是说，当前文章域的完整状态链路不是单点写入，而是：

1. 业务动作写关系事实
2. Redis 先承接热点增量
3. 查询时做数据库基础值 + Redis 增量叠加
4. 后续批量回刷数据库或同步 ES

这也是为什么文章域读起来会明显比普通模块更“系统化”。

| `/api/articles/favorite/contents` | DELETE | 从收藏夹批量移除文章 |
| `/api/articles/history` | GET | 浏览历史列表 |
| `/api/articles/history` | DELETE | 删除浏览历史 |
| `/api/articles/category` | GET/POST/DELETE | 分类管理 |
| `/api/articles/category/options` | GET | 分类下拉选项 |
| `/api/articles/tags` | GET/PUT/DELETE | 标签管理 |
| `/api/articles/tags/options` | GET | 标签下拉选项 |
| `/api/articles/top` | GET/POST/DELETE | 置顶能力 |

### 9.15 当前边界

- 当前文章详情不负责浏览统计
- 当前正文只维护安全 `content`
- 当前搜索即时同步和 Binlog 同步并存，不是纯业务双写，也不是纯异步唯一链路

## 10. 业务模块：评论系统

### 10.1 模块定位

评论系统当前明确只做两级嵌套：

- 一级评论
- 二级评论

这不是“还没来得及做无限级”，而是明确取舍后的结果。

### 10.2 为什么只做两级

无限级评论树在内容平台里通常会带来三类复杂度：

1. SQL 和分页复杂
2. 缓存与计数维护复杂
3. 前端展示和交互复杂

当前项目很清楚地选择了“两级就够用”的方案：

- 一级评论是主评论
- 二级评论是对一级评论或其他二级评论的回复
- 即使回复二级评论，也仍归属同一个 `root_id`

这让后端模型和前端展示都更可控。

### 10.3 评论列表的个性化增强

评论列表当前已经不只是公共内容列表，而是支持“游客访问 + 登录增强”。

当前一级评论和二级评论列表，在有登录态时都会额外返回：

- `is_digg`
- `relation`

其中：

- `is_digg` 表示当前查看者是否点赞过该评论
- `relation` 表示当前查看者与评论作者的关系

这种设计很实用，因为它让前端评论区可以直接做：

- 点赞按钮高亮
- 好友 / 粉丝标识
- 进一步互动入口

而不需要再发一堆补充请求。

### 10.4 评论管理列表

评论管理列表是另一条读视图：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/comments/man` | GET | 评论管理列表 |

当前支持三种模式：

- `type=1`：查我文章下的评论
- `type=2`：查我发的评论
- `type=3`：管理员查所有评论

其中 `type=1` 还有一个很有业务味道的细节：  
它会返回文章作者与评论人的 `relation`。  
也就是说，这个 `relation` 是站在“文章作者视角”计算的，而不是站在游客视角。

### 10.5 点赞、回复数与缓存叠加

评论列表返回时，当前会把 Redis 中的增量值叠加进来：

- 点赞数增量
- 回复数增量

这与文章计数设计保持一致，说明项目在评论域里也统一采用了：

- 数据库基础值
- Redis 热增量
- 查询时叠加

### 10.6 评论创建与回复链路

评论创建当前既支持：

- 文章一级评论
- 对一级评论的回复
- 对二级评论的回复

但不管回复的是一级还是二级，系统都会把它们归到同一个 `root_id` 下面。  
这就是当前“两级评论树”实现的关键：

- 公开展示只需要一级和二级两层
- 回复语义仍能通过 `reply_id` 保留

也就是说，系统没有做无限树，但保留了“回复给谁”的语义。

### 10.7 删除评论

删除评论时，当前不仅删评论本身，还会同步回滚：

- 文章评论数缓存
- 根评论回复数缓存
- 评论点赞缓存

这是一个很重要的细节。  
它说明删除动作在当前项目里不是“删记录”，而是“清整条关联链路上的派生状态”。

### 10.8 接口总表

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/comments` | GET | 一级评论列表 |
| `/api/comments/replies` | GET | 二级评论列表 |
| `/api/comments/man` | GET | 评论管理列表 |
| `/api/comments` | POST | 发表评论或回复 |
| `/api/comments/:id/digg` | POST | 点赞切换 |
| `/api/comments/:id` | DELETE | 删除评论 |

### 10.9 当前边界

- 当前没有“一级评论自带热门二级评论摘要”能力
- 当前评论管理列表不会像公开评论列表那样统一补所有个性化字段

## 11. 业务模块：站内消息、全局通知与私信

### 11.1 站内消息

站内消息更偏业务事件提醒，例如：

- 评论提醒
- 回复提醒
- 点赞提醒
- 收藏提醒
- 审核相关提醒

当前它有两个显著设计点：

1. 用户可配置消息开关
2. 业务动作后的消息常走异步入库

这说明站内消息模块被设计成“业务提醒系统”，而不是单纯通知表。

主要接口包括：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/sitemsg` | GET | 消息列表 |
| `/api/sitemsg` | POST | 标记已读 |
| `/api/sitemsg` | DELETE | 删除消息 |
| `/api/sitemsg/conf` | GET | 获取消息配置 |
| `/api/sitemsg/conf` | PUT | 更新消息配置 |
| `/api/sitemsg/user` | GET | 获取用户消息摘要 |

当前站内消息还有几个值得单独记录的设计点：

#### 消息开关不是全局一个布尔值

当前用户配置里至少区分了：

- 点赞通知
- 评论通知
- 收藏通知
- 私信通知

这说明消息系统的设计目标不是“给用户一个总开关”，而是允许用户按消息类型细粒度控制。

#### 消息列表本身带类型分区

`GET /api/sitemsg` 当前通过 `t` 字段区分：

- `1` 评论和回复
- `2` 点赞和收藏
- `3` 系统通知

这说明站内消息并不是一张无差别混排的大列表，而是从接口层就允许前端按业务语义分组展示。

#### 用户消息摘要是为了首页角标而设计

`GET /api/sitemsg/user` 返回的不是明细，而是：

- 评论消息数
- 点赞 / 收藏消息数
- 私信消息数
- 系统通知数

这条接口的设计目标非常明确，就是服务导航栏、消息中心入口和首页角标。

#### 站内消息为什么更像“业务事件投递层”

站内消息并不是后台手工创建的公告列表，而是由业务动作触发的提醒结果。  
常见触发源包括：

- 文章被评论
- 评论被回复
- 文章被点赞
- 评论被点赞
- 文章被收藏
- 系统审核或系统级消息

因此它的核心价值不是“存消息文本”，而是让业务动作在不直接耦合前端展示的前提下，能够投递到用户消息中心。

### 11.2 全局通知

全局通知和站内消息不同，它更像平台公告。  
当前支持：

- 管理员发布
- 用户查看
- 用户标记已读
- 用户删除自己的可见记录

管理员接口还接了审计日志采集，这说明全局通知被视为后台重要动作。

主要接口包括：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/global_notif` | GET | 全局通知列表 |
| `/api/global_notif/read` | POST | 用户标记已读 |
| `/api/global_notif/user` | DELETE | 用户删除自己的通知记录 |
| `/api/global_notif` | POST | 管理员发布全局通知 |
| `/api/global_notif` | DELETE | 管理员删除全局通知 |

全局通知与站内消息最大的不同，在于它处理的是“平台面向一批用户的公告”，而不是“某个业务事件对某个用户的提醒”。  
因此它额外需要解决两类问题：

#### 可见性规则

管理员创建通知时，不只是填写标题和正文，还会带：

- 过期时间
- 用户可见规则
- 图标
- 跳转链接

这意味着一条全局通知从设计上就是“可运营的公告对象”，而不只是纯文本消息。

#### 用户侧状态分离

用户是否已读、是否删除自己的通知记录，并不直接改公告主表。  
这类状态更适合放在用户侧关系表中维护，避免平台公告对象被每个用户的阅读状态污染。

#### 全局通知的两种读视图

全局通知当前至少有两种不同语义的列表查询：

- 用户视角：查“我当前还能看到哪些通知、哪些已读”
- 管理员视角：查“平台发布过哪些通知、状态如何”

因此同一资源对象会有不同读模型，这是全局通知和普通业务消息非常不同的一点。

### 11.3 私信

私信模块当前已经不是预留状态，而是完整接了：

- 会话列表
- 消息列表
- 已读
- 用户级删除
- WebSocket 鉴权票据
- WebSocket 通信入口

设计上，它采用会话和消息两层模型：

- 会话保存摘要和用户级状态
- 消息保存正文和阅读态

同时，发送私信还会受到用户关系与隐私配置约束，这点让私信不至于成为完全开放的骚扰入口。

主要接口包括：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/chat/sessions` | GET | 会话列表 |
| `/api/chat/sessions` | DELETE | 用户删除会话 |
| `/api/chat/messages` | GET | 消息列表 |
| `/api/chat/messages` | DELETE | 用户删除消息 |
| `/api/chat/read` | POST | 标记消息已读 |
| `/api/chat/ws-ticket` | GET | 获取 WebSocket 票据 |
| `/api/chat/ws` | GET | WebSocket 实时通信入口 |

当前私信模块还有几处非常体现工程取舍的设计：

#### 会话按“用户视角”存两份

一段对话不会只存一条共享会话，而是双方各保留一条会话记录。  
这样每一侧都可以独立拥有：

- 未读数
- 删除状态
- 置顶状态
- 静音状态

这比“所有人共享一条会话记录”更适合做用户级会话列表。

#### 消息删除是用户级删除

当前删除消息并不是把消息主记录直接物理删除，而是借助用户态表记录“某个用户看不到这条消息了”。  
这能避免一方删除影响另一方历史。

#### WebSocket 不直接裸连

系统先通过 `/api/chat/ws-ticket` 获取短期票据，再进入 `/api/chat/ws`。  
这种做法比直接把长期登录态暴露在 WebSocket 握手上更稳，也更方便做短生命周期鉴权。

#### 私信不是完全开放入口

私信还会受到：

- 好友 / 关注关系
- 用户隐私配置
- 陌生人私信开关
- 发送频率限制

的共同约束。  
这说明项目已经把私信当成一个需要治理的社交入口，而不是随便开个聊天通道。

#### 消息类型与内容约束

当前私信消息不是只有纯文本，至少已经支持：

- 文本
- 图片
- Markdown

这说明私信模块不是一个“最低配 IM 实验”，而是已经考虑到站内内容表达能力。  
同时，不同类型消息在发送前通常还会经过：

- 格式归一化
- 内容长度限制
- 权限和配额校验

#### 已读与未读设计

当前已读能力并不是“拉消息列表就默认全部已读”，而是独立通过接口推动。  
这样做的好处是：

- 前端可以更清晰地控制已读时机
- WebSocket 推送和列表查询不会互相干扰
- 未读数可以和会话摘要更稳定地协同

### 11.4 当前边界

- 当前消息系统已较完整，但仍主要服务站内闭环，没有进一步做推送网关集成
- 当前私信的实时入口已接入，但更高级的消息投递与离线补偿策略仍有演进空间

## 12. 业务模块：用户关系

### 12.1 模块定位

用户关系模块当前聚焦于关注关系，而不是完整社交系统。  
当前关系状态有四种：

- 陌生人
- 已关注
- 粉丝
- 好友

如果把它们用更明确的语义写出来，可以理解为：

| 值 | 语义 |
| ---- | ---- |
| `1` | 陌生人，双方都未形成关注 |
| `2` | 我已关注对方，但对方未关注我 |
| `3` | 对方关注了我，但我未关注对方 |
| `4` | 双向关注，即好友 |

这套关系态虽然不复杂，但已经足以支撑：

- 主页关系标识
- 评论区关系标识
- 粉丝 / 关注列表的关系字段
- 私信可达性判断

它其实表达的是一种很务实的社交关系模型：

- 不追求复杂的社交图谱
- 只解决内容平台最常见的互动判断
- 让“能不能私信、主页显示什么关系、评论区挂什么标识”都有统一答案

### 12.2 关注 / 取关

关注和取消关注当前是两条独立接口，而不是切换接口：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/follow/:id` | POST | 关注用户 |
| `/api/follow/:id` | DELETE | 取消关注 |

这种设计的好处是：

- 语义明确
- 不需要前后端猜当前状态
- 业务错误边界更清楚

此外，独立的关注和取关接口也更适合配合审计、限流和异常处理。  
如果使用切换式接口，排查“为什么变成当前状态”通常会更难。

### 12.3 粉丝数与关注数的冗余维护

关注关系当前不只写 `user_follow_models`，还会在事务中同步更新：

- 关注者的 `FollowCount`
- 被关注者的 `FansCount`

这体现了当前系统对高频主页统计的总体思路：

- 事实表记录真实关系
- 冗余表负责高频展示值

这里真正重要的点在于“事务内同步”。  
只有这样，系统才能尽量避免出现下面这种短时脏状态：

- 关系表已经插入，但粉丝数没变
- 或者粉丝数加了，但关系表实际没写成功

### 12.4 关注列表与粉丝列表

当前两个列表都要求登录：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/follow` | GET | 关注列表 |
| `/api/fans` | GET | 粉丝列表 |

二者都支持：

- 默认查自己
- 在目标用户公开时查别人
- 返回 `relation`

这意味着它们本身已经是“列表 + 当前关系态”的复合接口，而不是只返回静态用户卡片。

从返回结构看，这两个列表除了昵称和头像，还会返回：

- 简介
- 关注时间
- 当前关系态

这使它们既能服务个人主页关注 / 粉丝页，也能服务“从列表里直接发起互动”这种前端需求。

### 12.5 关系计算能力

当前系统内部提供：

- `CalUserRelationship`
- `CalUserRelationshipBatch`

批量版本尤其重要，因为很多接口都要同时给列表项补关系态，例如：

- 关注列表
- 粉丝列表
- 用户主页
- 评论列表

如果每一项都单独查两次关注关系，代价会非常高。  
因此批量关系计算是当前用户关系模块非常核心的一块基础能力。

从系统视角看，关系计算已经不再是关注模块自己的私有逻辑，而是多个读接口共同依赖的基础设施：

- `users/base` 需要给主页补 `relation`
- `follow` / `fans` 列表要给每一项补 `relation`
- 评论列表需要给评论作者补 `relation`
- 评论管理列表在“看我文章下评论”时也要补 `relation`

这就是为什么关系批量计算值得单独沉淀成 service，而不是散落在多个 handler 里。

### 12.6 可见性与权限边界

关系列表并不是完全公开的。  
当前至少会受到这些因素影响：

- 是否登录
- 目标用户是谁
- 目标用户是否公开了关注 / 粉丝列表

这意味着关系模块虽然简单，但已经开始承担隐私边界，而不是一个无权限约束的公开社交图。

### 12.7 当前边界

- 当前没有更复杂的社交关系模型
- 当前也没有对关注 / 取关做严格风控限额

### 12.8 关系模块接口总表

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/follow/:id` | POST | 关注用户 |
| `/api/follow/:id` | DELETE | 取消关注 |
| `/api/follow` | GET | 关注列表 |
| `/api/fans` | GET | 粉丝列表 |

## 13. 业务模块：搜索、AI 与统计

### 13.1 搜索模块

#### 模块定位

搜索模块当前主要开放文章搜索：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/search/articles` | GET | 文章全文搜索 |

#### 设计思路

当前搜索采用三段式读模型：

1. **ES 召回**
   - 关键词匹配
   - 标签过滤
   - 排序
   - 高亮和分段命中
2. **MySQL 回表**
   - 查真实业务字段
   - 补作者、分类、标签等
3. **Redis 叠加**
   - 补最新点赞 / 收藏 / 浏览 / 评论增量

设计重点不是“让 ES 直接返回页面”，而是把：

- 搜索能力
- 业务真相
- 实时计数

分别交给最擅长的组件。

接口参数层面，文章搜索当前也已经不是只有一个 `key`。  
它还支持：

- `type`：普通搜索、猜你喜欢、作者文章、自己文章、管理员搜索
- `sort`：默认、最新、最多回复、最多点赞、最多收藏、最多浏览
- `tag_list`
- `category_id`
- `user_id`
- `top_search`
- `status`

这说明当前搜索模块既承担“公开站内搜索”，也承担了若干后台和作者视角的过滤读能力。

#### 搜索结果结构为什么和文章列表相近

当前搜索返回的结果并不是只给一组“标题 + 片段”。  
它会尽量对齐文章列表常用字段，包括：

- 标题、摘要、封面
- 浏览 / 点赞 / 评论 / 收藏数
- 标签
- 作者昵称、头像
- 分类标题
- 置顶信息

这样做的好处是：

- 前端搜索页可以复用文章卡片组件
- 搜索和普通文章流的展示语义更接近
- 计数补偿逻辑可以复用文章域实现

#### 为什么搜索结果还要回表

ES 可以给出很好的召回和相关度排序，但它不是业务真相。  
回表的价值主要体现在：

- 以 MySQL 为准拿文章状态
- 统一补作者、分类、标签
- 统一叠加 Redis 热增量
- 让接口返回结构和文章列表 / 文章详情域更一致

从维护角度讲，这能显著降低“ES 文档和真实业务状态逐渐漂移”的风险。

#### `top_search` 的含义

搜索接口里的 `top_search` 并不是普通布尔筛选，而更像“是否在搜索排序中显式考虑置顶优先”。  
这说明搜索结果也在努力和站内内容展示秩序保持一致，而不是完全只看 ES 的相关度分数。

#### 同步设计

ES 当前不是完全业务双写，也不是完全只靠定时任务。  
它采用的是混合方案：

- 主要同步链路依赖 MySQL -> ES 同步机制
- 某些业务变更场景下会补局部刷新

这是一种在工程复杂度和搜索时效性之间的折中。

### 13.2 AI 模块

#### 模块定位

AI 模块当前不是一个统一聊天框，而是一组围绕文章写作和站内搜索的辅助能力：

- 元信息提取
- 全文评分
- 选区改写
- 选区诊断
- AI 搜索

#### 统一基础层

当前所有 AI 能力都复用：

- 配置校验
- HTTP 请求封装
- 非流式 Chat
- 流式 ChatStream

这让业务层只关心：

- 提示词设计
- 输入裁剪
- 输出归一化

这意味着 AI 模块本质上是“业务场景层 + 通用 AI 基建层”的组合。  
当前项目没有把所有提示词和 HTTP 调用散落在 handler 中，而是已经形成了相对清晰的复用结构。

#### 元信息提取

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/ai/metainfo` | POST | 标题、摘要、分类、标签建议 |

设计重点是：

- 分类从当前用户已有分类候选里选
- 标签从系统已启用标签里选
- AI 返回后还要再次归一化

这说明它不是“让 AI 自由写”，而是让 AI 在业务候选集内做推荐。

换句话说，AI 元信息提取更像“在已有业务约束内做辅助填充”，而不是让大模型直接决定站点分类和标签体系。

#### 全文评分

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/ai/scoring/article` | POST | 对整篇文章做评分与问题分析 |

设计重点是：

- 短文直接一次评分
- 长文分段评分并累积状态
- 最终返回结构化结果而不是一句模糊评价

这说明评分模块更像“写作体检系统”。

从设计取舍上看，全文评分之所以做成长文分段，不只是为了绕过 token 限制，还因为：

- 长文一次喂给模型不稳定
- 分段后更容易定位问题段落
- 最终结构化输出更利于前端展示维度分

#### 选区改写与诊断

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/ai/overwrite` | POST | 选区改写，SSE 返回 |
| `/api/ai/diagnose` | POST | 选区诊断，SSE 返回 |

两者都不是处理全文，而是针对编辑器选区。  
因此都会带：

- 选中文本
- 选区前文
- 选区后文
- 文章标题

这让 AI 能在局部写作场景下仍然理解上下文。

这种输入结构说明，当前 AI 改写 / 诊断并不是“对孤立文本做润色”，而是尽量模拟编辑器里的真实写作上下文。

#### AI 搜索

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/ai/search/list` | POST | 搜索意图改写后返回文章列表 |
| `/api/ai/search/llm` | POST | 搜索意图识别 + AI 总结结果，SSE 返回 |

当前 AI 搜索不是让大模型直接编文章，而是：

1. 先识别是否为搜索意图
2. 提炼结构化搜索条件
3. 调用站内搜索拿真实文章结果
4. 再让 AI 生成自然语言总结

这个设计非常正确，因为它把：

- 内容检索
- 自然语言理解
- 结果组织

分成了不同层次，而不是交给模型一把梭。

AI 搜索这套设计尤其重要，因为它刻意避免了两个常见问题：

1. 模型胡乱编造站内并不存在的内容
2. 模型绕过站内搜索逻辑直接给出不受控答案

当前项目选择的是：

- 让模型理解用户意图
- 让站内搜索负责真实内容召回
- 再让模型基于真实召回结果组织总结

这是非常稳的路线。

### 13.3 数据统计模块

#### 模块定位

数据统计当前服务后台看板，不是开放给普通用户的数据大屏。  
主要能力包括：

- 汇总统计
- 近 7 天趋势
- 近 12 个月发文趋势

接口包括：

| 接口 | 方法 | 说明 |
| ---- | ---- | ---- |
| `/api/data/sum` | GET | 汇总统计 |
| `/api/data/growth` | GET | 近 7 天增长趋势 |
| `/api/data/article-year` | GET | 近 12 个月发文趋势 |

#### 汇总统计设计

`/api/data/sum` 当前返回：

- `flow_count`
- `user_count`
- `article_count`
- `message_count`
- `comment_count`
- `new_login_count`
- `new_sign_count`

这些指标并不是从一个统一统计表里出来，而是按指标特点分源读取：

- Redis：流量
- MySQL：用户、文章、消息、评论、今日注册
- ClickHouse：今日去重登录人数

这个设计很务实，先满足后台看板，不提前做复杂数仓。

也正因为如此，当前统计模块本质上是“聚合接口层”，而不是“统一统计数据仓库”。  
它做的事情是：

- 按指标选择最合适的数据源
- 把时间窗口和返回格式统一
- 让前端后台看板不需要自己拼多个接口

如果把汇总指标再按数据来源理解，会更清楚：

| 指标 | 主要来源 |
| ---- | ---- |
| `flow_count` | Redis 站点流量计数 |
| `user_count` | MySQL 用户表 |
| `article_count` | MySQL 文章表 |
| `message_count` | MySQL 消息相关表 |
| `comment_count` | MySQL 评论表 |
| `new_login_count` | ClickHouse 登录事件日志 |
| `new_sign_count` | MySQL 今日注册数 |

#### 近 7 天趋势设计

`/api/data/growth` 当前支持：

- `type=1` 流量
- `type=2` 发文
- `type=3` 注册

设计重点：

- 时间窗口固定 7 天
- 会补齐缺失日期
- 返回 `growth_num` 和 `growth_rate`

这使得前端图表不用自己再补日期洞。

这里还有一个很实际的设计点：  
增长率接口不只返回折线图数据，还额外返回：

- `growth_num`
- `growth_rate`

这让前端既能画图，也能在卡片上直接显示“较前一周期增加多少、增长百分比多少”。

#### 近 12 个月发文趋势

`/api/data/article-year` 只关心一个问题：  
最近 12 个月每个月发了多少篇**已发布文章**。

设计重点：

- 固定 12 个月窗口
- 缺失月份补 0
- 日期格式为 `YYYY-MM`

这条接口是典型的后台经营趋势接口，而不是通用统计引擎。

### 13.4 搜索、AI 与统计设计补充

把这三块放在同一章，不只是因为它们都偏“增强能力”，还因为它们都遵循同一个总原则：

- 不取代 MySQL 业务真相
- 只在自己擅长的维度增强系统

具体来说：

- 搜索增强的是全文召回和排序
- AI 增强的是创作和搜索理解
- 统计增强的是后台经营视图

因此三者虽然复杂，但都没有反过来篡夺主业务模型的控制权。

| `/api/data/growth` | GET | 近 7 天趋势 |

## 14. 业务模块：命令行、Global 与工具层

### 14.1 命令模块

命令模块当前承担离线运维和初始化职责。  
它的意义不是“提供一个方便玩的 CLI”，而是把这类能力从 Web 服务主链路中剥离出来。

当前能力包括：

- 数据库迁移
- ES 初始化
- ES 全量文章同步
- 交互式创建用户

这个设计让部署和初始化更清楚，也减少了后台暴露超级能力的风险。

如果从当前代码的 `flags` 实现看，命令模块已经形成了比较清晰的职责边界：

| 命令域 | 子命令 | 真实用途 |
| ---- | ---- | ---- |
| `-db` | 无 | 执行数据库迁移 |
| `-es` | `init` | 初始化 / 删除文章索引和流水线 |
| `-es` | `article-sync` | 按批次全量同步文章文档 |
| `-t user` | `create` | 交互式创建用户 |

这说明命令层不只是“开发调试便利入口”，而是实际承担了部分部署和数据治理职责。

从结构上看，当前命令模块已经形成了两种典型用法：

- 布尔型入口：如 `-db`
- 能力域 + 子命令：如 `-es -s init`

这说明项目在保持实现简单的同时，已经开始为后续命令扩展留出空间。

### 14.2 Global 模块

`global/` 是当前项目的全局依赖容器。  
它本身不承载业务逻辑，但几乎所有模块都会依赖它：

- `Config`
- `Logger`
- `DB`
- `Redis`
- `ClickHouse`
- `ESClient`
- `ImageCaptchaStore`
- `Flags`

设计思路很简单：  
项目没有上依赖注入框架，那就用最直接的全局单例方式把关键资源共享出去。

如果从“全局变量为什么值得单独成章”来看，原因在于几乎所有模块都会围绕它们组织依赖：

- handler 通过 `global.DB`、`global.Redis` 读写业务
- 中间件通过 `global.Logger`、`global.Config` 采集和校验
- 离线命令通过同一套全局初始化共享资源

也正因为如此，`global/` 同时带来了便利和耦合：

- 便利在于单体开发速度快
- 代价在于初始化顺序和测试隔离要更小心

当前比较关键的全局对象大致可以这样理解：

| 全局对象 | 主要作用 |
| ---- | ---- |
| `Config` | 全局配置镜像 |
| `Flags` | 当前进程启动参数 |
| `Logger` | 统一日志出口 |
| `DB` | MySQL 业务真相存储入口 |
| `Redis` | 风控、去重、计数与上传状态入口 |
| `ClickHouse` | 日志分析查询入口 |
| `ESClient` | 搜索索引与文档同步入口 |
| `ImageCaptchaStore` | 图片验证码存储入口 |

### 14.3 工具层的角色

当前工具层虽然不像业务模块那样显眼，但其实承担了很多关键工作，例如：

- JWT 工具
- Markdown 安全处理
- 请求元信息提取
- IP 归属地解析
- 日志安全序列化

这说明项目已经把不少容易散落在业务里的横切逻辑沉淀到了工具层，而不是让它们重复出现在多个 handler 里。

从长期维护角度看，工具层最重要的价值不是“省几行代码”，而是把这些规则统一起来：

- Markdown 的安全清洗不要每个模块各写一套
- JWT 的 claims 结构和签发逻辑不要四处分散
- 日志脱敏不要依赖调用方自觉
- IP 与地址解析不要让多个模块各自接不同来源

这些事情一旦分散，后续升级会非常痛苦。当前工具层的存在，本质上是在避免这种分裂。

另外，工具层与中间件层的边界也值得说明：

- 工具层更偏无状态能力和纯函数
- 中间件层更偏请求生命周期中的拦截与上下文注入

把这两层区分开，代码会更容易维护。

## 15. 部署与运维视角

### 15.1 部署形态

当前项目仍然是单体部署形态，但依赖的基础设施并不少：

- MySQL
- Redis
- Elasticsearch
- ClickHouse
- Fluent Bit
- 七牛云

部署资源当前主要沉淀在 `init/deploy/` 中。

如果按部署资源目录看，当前仓库已经把一套相对完整的本地 / 服务器部署素材沉淀下来了：

- `blog_server/`：后端配置样例
- `blog_web/`：前端静态资源
- `sql_master/`、`sql_slave/`：MySQL 主从相关配置
- `redis/`：Redis 配置与持久化目录
- `es/`：ES 容器与数据目录
- `clickhouse/`：ClickHouse 初始化 SQL
- `fluent-bit/`：日志采集配置
- `nginx/`：反向代理与证书目录
- `docker-compose.yml`：整套依赖的编排入口

这说明项目虽然是单体业务，但部署视角已经不是“启动一个二进制就完事”，而是一个依赖多组件协同的增强型单体。

### 15.2 配置文件与部署素材的角色

从当前仓库结构看，部署并不只依赖一个 `settings.yaml`。  
至少有这几类配置共同参与：

- 主应用配置
- 站点默认运行时配置 `site_default_settings.yaml`
- Nginx 反向代理配置
- Fluent Bit 采集配置
- ClickHouse 初始化 SQL
- MySQL / Redis / ES 容器配置

这说明部署问题很多时候不是“后端代码有 bug”，而是多份配置之间的协同问题。

### 15.3 推荐理解成“两类依赖”

从运维角度，当前依赖其实可以分为两类：

#### 核心业务依赖

- MySQL
- Redis
- 七牛云

没有这几项，站点基本无法正常提供主业务。

#### 增强能力依赖

- Elasticsearch
- ClickHouse
- Fluent Bit
- Nginx / HTTPS
- Binlog 监听所需主从能力

这些依赖未必会让进程完全起不来，但会让搜索、日志、资源治理、回调安全和实时能力显著退化。

### 15.4 为什么日志链路要一起部署

如果只部署应用，不部署：

- Fluent Bit
- ClickHouse

那么后台日志查询模块就无法完整工作。  
这说明日志系统在当前项目里已经不是“可有可无的附加件”，而是后台管理的一部分。

进一步说，日志链路当前是一个闭环：

1. 应用写本地结构化 JSON 日志
2. Fluent Bit 监听日志目录
3. Fluent Bit 解析并写入 ClickHouse
4. 后台日志接口再从 ClickHouse 查询

缺其中任何一环，后台日志模块都会退化成“只能看应用控制台”，这和当前系统的设计目标是不一致的。

### 15.5 HTTPS 的必要性

当前建议部署时使用 HTTPS，原因不仅仅是浏览器要求。  
它会直接影响：

- 登录态安全
- 七牛回调
- WebSocket
- AI 接口请求

对于一个已经接入后台管理、对象存储回调和实时通信的系统来说，HTTPS 已经属于基础配置，而不是高级增强。

从当前项目能力来看，HTTPS 主要在四个地方起实际作用：

1. 登录 token、Cookie、鉴权头部的传输安全
2. 七牛上传 / 审核回调的外网可信接入
3. WebSocket 实时私信连接
4. 后台管理和 AI 请求的外网访问安全

### 15.6 推荐部署顺序

如果要把当前系统从零部署起来，一个更符合现有设计的顺序通常是：

1. 准备 MySQL、Redis、Nginx、后端配置
2. 执行 `-db` 数据库迁移
3. 启动后端，确认基础 API 可用
4. 初始化 ES 索引与流水线
5. 执行 `-es -s article-sync` 做一次全量同步
6. 部署 ClickHouse + Fluent Bit，确认日志能入库
7. 打开图片直传、审核回调、Binlog 监听
8. 最后再验证 AI、WebSocket、后台统计等增强能力

这样做的好处是：  
先让主业务站起来，再逐步打开增强能力，排障时更容易收敛。

### 15.7 运行校验与排障入口

当前系统部署完成后，比较合理的验收顺序通常是：

1. 访问基础公开接口，确认 HTTP 与数据库可用
2. 登录后台，确认认证、验证码、会话链路可用
3. 试发一篇文章，确认文章、分类、标签、审核链路可用
4. 搜索文章，确认 ES 同步与查询正常
5. 查看后台日志，确认 Fluent Bit 与 ClickHouse 正常
6. 上传图片，确认七牛回调、审核回调与图片引用正常
7. 测试私信和 WebSocket，确认实时链路可用

如果部署后出现问题，也可以按模块逆推：

- 文章看不到：先查文章状态、审核规则、ES 同步
- 图片异常：先查七牛配置、回调地址、审核状态、引用河流
- 日志后台空白：先查 Fluent Bit、ClickHouse、日志目录
- 私信不通：先查 ticket、WebSocket、关系与私信开关
